# 氛寸（Fēn Cùn）· 产品最终优化方案（定稿）

> 本文档由首席产品架构师综合五位专家草稿与红队审查意见整合而成，可直接用于作品集叙事、面试讲解与落地开发。全程对齐已本地核实的 ledecanteur 真实数据集，严守四条戒律：不伪精确、不过度设计、轻冷启动、有反馈闭环。规模严格控制在“一个全栈个人几周内可交付”。

## 1. 产品定位

**一句话定位**：氛寸——你已经有这些香水，它帮你决定此刻该喷哪一瓶、怎么喷得恰到好处。

**传播版**：别人帮你挑香水，氛寸帮你用好香水。

**三段式展开**：

- **是什么**：一个香水“使用决策”智能体。它接管的不是“买什么”，而是用户拥有香水之后每天真实发生的动作——站在香水柜前纠结今天喷哪瓶、喷多少、合不合时宜。它做三件不可拆分的事：① **推荐**（你的香水库 × 实时情境 → 此刻最适合的一瓶）；② **可更换**（一键换成你想喷的任意一瓶，决策权永远在你手里）；③ **教你怎么用**（核心壁垒：喷几下、喷在哪、社交距离、留香区间、是否合时宜、有无风险、换香后如何调整）。

- **不是什么**：不是导购/电商（不挂商城、不种草，用户来时已拥有香水）；不是香评社区/百科（香水时代已做得好，不重复）；不是 AI 调香（嗅氪的方向，我们让你用好已有的瓶子）；不是带滑块的查询工具（情境自动感知，输出是“懂的人会对你说的话”）。

- **为谁**：已拥有 ≥3 瓶香水、却常“喷错/喷多/喷得没存在感/不知道今天用哪瓶”的人。

**边界澄清（采纳红队①）**：氛寸**全程不碰交易**。长期变现走“卖分寸本身”——订阅高级版（更细个人香水画像、年度用香回顾、季节轮换报告），而非“按情境推荐你库里缺的那一类”那种绕圈导购。嘴上戒导购、路线图却带货，是最易被面试官看穿的自欺，已彻底删除。

## 2. 产品世界观

**它相信什么**：香水的意义不在“拥有多少”，而在“此刻用得对不对”。同一瓶 Bergamot（佛手柑），在 35℃ 盛夏正午和零度深夜地铁里是两种东西。决定它好不好的，从来不是它本身，而是它与天气、体感、场合、与你今天想成为谁之间的关系。香水是**使用的艺术**，而非收藏的军备竞赛。

**它反对什么**：

| 反对的旧观念 | 氛寸的主张 |
|---|---|
| 越浓越高级 | 扩散有边界，过浓是失礼；高级是“刚好被闻到” |
| 越多越好（不停买更贵的） | 你已拥有的大多没用对，先把手里的用到位 |
| 香水是“买”的学问 | 香水是“用”的学问；买只发生一次，用发生每一天 |

**品牌内核——分寸感**（三重同一）：气味的分寸（喷多少、喷在哪、扩散几米）＋ 场合的分寸（配不配得上此刻的你、房间、人）＋ 表达的分寸（香水是你今天无声说出的第一句话）。

> 一句立住世界观：**氛寸不教你买香水，氛寸教你“懂分寸地用香水”。**

## 3. 核心用户与使用场景

### 3.1 三画像（诚实标注真实频率，采纳红队②③）

> 共同硬门槛：已拥有 ≥3 瓶香水——拥有量越高、决策成本越高、价值越大。

- **画像A · 纠结的轮换者（唯一日活基本盘）**：26–35 岁，一二线，5–15 瓶，1/3 常年吃灰。**真实价值不在“解决柜前纠结”（弱痛点，他无脑喷老相好且不痛苦），而在“帮他翻出吃灰的好瓶子”＋“天气突变时提醒他常喷那瓶今天会翻车”**——这种“它发现了我没意识到的问题”才是真钩子。

- **画像B · 怕出错的场合敏感者（事件触发型，低频，决定口碑）**：职场新人/商务人士，3–8 瓶。香水是社交工具不是爱好，最怕“用得不得体”。**诚实承认：他不是每天打开，是“重要场合才打开”**。但“幸亏没喷那瓶”一次体验就形成依赖，传播力强。

- **画像C · 钻研型爱好者（少数派，决定专业可信度与反馈质量）**：15+ 瓶。懂 sillage/longevity，是可信度试金石，糊弄不了。最优质的反馈来源。

> **取舍（修正后）**：日活只押 A；B 是事件触发的口碑放大器、不计入高频；C 是专业背书与反馈飞轮的少数派。面试时主动说“我的高频盘其实窄”，比被问穿强十倍。

### 3.2 高频使用场景剧本

| # | 场景 | 氛寸介入 | 输出示例 |
|---|------|---------|---------|
| S1 | 早间通勤前（每日触发） | 打开即见今日推荐 | “今天上海 28℃、湿度 80%、闷。从你的库里选**蓝风铃**：清爽柑橘适合这种黏腻天。喷 2 下（手腕＋颈侧），别上身。今天它属于‘走近才闻到’的距离。” |
| S2 | 重要场合临时确认 | 切到“场景：商务会面” | “你选的 Aventus（信仰之水）在商务场合偏稳。但今天室内空调密闭，建议只喷 1 下、喷在衣领内侧。” |
| S3 | 换香后不确定用法 | 一键换香 | “换成 Tobacco Vanille（烟草香草）没问题。提醒：它扩散偏外放、偏甜偏厚，今天 30℃ 建议减到 1 下、只喷手腕。” |
| **S4** | **天气突变预警（主打钩子）** | 情境刷新自动提示 | “今天骤降到 12℃ 还下雨——你常喷的清新柑橘会被压住、留不住。库里**Spicebomb（香料炸弹）**更扛得住冷湿天，要不要换？” |
| **S5** | **吃灰提醒（主打钩子）** | 合适情境下顶出冷落瓶 | “这瓶你 35 天没碰了——今天干冷的天气正是它的主场，要不要翻出来？” |
| S6 | 用后复盘（反馈闭环） | 晚间轻问一句 | “今天那瓶用着怎么样？〔刚好/偏淡/太冲〕”——一次点击，系统收敛明日推荐。 |

> **S4＋S5 是发现型价值，提到 MVP 必做、当传播主线**（采纳红队③）——它们让用户“咦一下”，比“今天喷哪瓶”更能立住需求。

## 4. 核心用户路径

### 4.1 首次冷启动（目标：3 分钟内建库＋看到第一个“懂我”的推荐）

```
① 落地页：首屏直接演示一张推荐卡（而非口号）
   ⚠ 讲不清价值就跳出 → 首屏即 demo

② 搜名秒添加（冷启动生命线，必须轻）
   输入"祖马龙蓝风铃" → 实时联想 → 点选即入库，不填任何卡片
   ⚠ 搜不到 → 中文别名/品牌映射 +「找不到?手动记下名字」兜底
   ⚠ 不要求填季节/喜好等，一律后置或自动推断

③ 自动获取情境（QWeather 静默拉取定位天气，不让用户填滑块）
   ⚠ 定位被拒 → 城市手选兜底，不卡死

④ 出第一张推荐卡（推荐＋理由＋用法三件套同屏）
   ⚠ 第一张卡英文出戏/理由空泛 = 直接判死 → 中文化与理由质量是生死线

⑤ 轻引导一次"换香"或埋"明早再来"钩子
```

### 4.2 日常回访（目标：5 秒完成今日决策，留一条反馈）

```
① 触达：早间打开即见今日推荐卡（每天天气在变=每天值得看）
   ⚠ 推荐每天雷同 → 情境差异必须体现在推荐与文案

② 看卡（推荐瓶＋一句话理由＋用法三件套）
   ⚠ 理由像模板复读 → 必须结合当日具体天气/场景

③ 认同→直接用；不认同→一键换香（换香后用法即时重算）

④ （可选）切换场景重算：通勤/会议/约会/聚会/正式/居家/运动（4–7个高频）

⑤ 用后反馈（晚间或下次打开补问）
   ⚠ 反馈了看不到变化 → 必须让反馈在后续推荐中可感生效
```

> **路径核心张力**：回访留存押在两个慢变量——**情境每天真实在变**（天气驱动，保证“每天不一样”）＋ **系统越用越懂我**（反馈驱动，保证“越来越准”）。缺一个就退化成小玩具。

## 5. Agent 能力设计

### 5.1 角色与边界

| 维度 | 定义 |
|---|---|
| 角色 | 用香决策顾问。不卖、不造，只回答“你已有的香水里此刻该用哪瓶、怎么用”。 |
| 输入 | 香水库 ＋ 实时情境 ＋ 历史反馈 ＋ 可选自然语言诉求 |
| 输出 | 主推一瓶 ＋ 备选 2–3 瓶 ＋ 用法 ＋ 风险提示 ＋ 一段自然语言解释 |
| 能力边界 | ① 只在用户**已有**库内推荐，绝不推荐没有的香水；② 不预测精确留香小时数；③ 不做医学/过敏诊断；④ 不调香、不评测优劣 |

### 5.2 混合架构：规则做决策，LLM 做表达（全方案最关键的一条戒律）

```
自然语言诉求 ──[DeepSeek 解析]──► 结构化情境补丁(JSON)
                                      │
传感情境(天气API) ────────────────────►├─► 合并为 Context
                                      │
香水库 × Context ──[规则引擎 scoreFn]──► 打分排序(确定性)
                                      │
Top-N + 用法计算(规则) ──[DeepSeek 解释]──► 自然语言文案
                                      ▼
                                Recommendation
```

| 任务 | 谁做 | 为什么 |
|---|---|---|
| 匹配打分、排序 | **规则引擎** | LLM 打分不可复现、不可审计；规则每一分可拆给用户看 |
| 喷量/部位/距离/风险判定 | **规则引擎** | 这是壁垒，交给 LLM 会幻觉（“喷膝盖后侧 3 下”） |
| 自然语言诉求→结构化情境 | **DeepSeek** | 规则无法穷举“约会/见家长/挤地铁”，正是 LLM 擅长 |
| 把打分结果翻译成有温度的解释 | **DeepSeek** | 让它像“懂香水的朋友”；但**解释必须基于规则给定的事实，不准 LLM 改结论** |

> **LLM 是“嘴”和“耳朵”，规则引擎是“大脑”。** 决策权在规则，表达权在 LLM。即便 DeepSeek 超时/限流，规则引擎仍能给推荐＋模板用法（降级），产品不白屏。**天气永远来自 QWeather，绝不让 LLM 编造天气。**

DeepSeek 解析强制 JSON（`response_format: json_object`）＋ zod 校验；解析失败则整段降级为空补丁，绝不阻断推荐。

## 6. 香水数据集调研与选择

**为什么选 ledecanteur（Fragrantica 社区抓取）**：氛寸的灵魂是“这瓶怎么用”，而“怎么用”依赖真实人群使用反馈而非厂商宣称。这套数据里 longevity/sillage/seasons/daypart/price_value 全是**社区投票计数**——“这瓶夏天合不合适”不是编辑拍脑袋，而是 `seasons.summer` 有多少人投票；“扩散强不强”是真实佩戴者打的 `sillage.average`(1–4)。厂商说“持久 8 小时”不可信，3000 人投票的均值可信。

**独特价值**：多维投票可算占比（不是非黑即白）；accords 带 0–100 强度；relation(have/had/want)＋popularity 可做搜索权重；similar 是天然的“换一瓶”备选来源，不必早期上向量库。

**局限与处理（如实承认）**：

| 局限 | 处理 |
|---|---|
| 全英文 | 三层中文映射（见下）。**这是体验生死线** |
| Fragrantica 口径偏欧美深度爱好者 | 季节/扩散用相对占比，不用绝对阈值定生死 |
| 投票噪声/长尾稀疏 | 见下“数据规模决策” |
| sillage 只有 1–4 四档 | 正好契合非伪精确，映射 4 个人话档 |
| 缺浓度(EDP/EDT) | 从 name 弱推断或标“未知”，不为补字段伪精确 |

**数据规模决策（采纳红队③，最重要的一砍）**：MVP **不入全量 36,705**。只精选 **Top 1,000–2,000 最热门香水**（按 popularity，中国用户真会拥有的），把这部分的 notes/brand/accords **100% 人工中文化**（2000 款香调去重后约一两千个词，一个人 2–3 天可啃完）。**用“小而全中文”换“大而露英文”，直接拆掉自己定义的生死线。** 搜不到的长尾老老实实让用户“手动记名字”。

**英文→中文三层映射**：
1. **accords 香调表（封闭集合，约 60–70 个，全量人工）**：citrus→柑橘、fresh→清新、white floral→白花、woody→木质、sweet→甜调、vanilla→香草、amber→琥珀、leather→皮革、tobacco→烟草、aquatic→水生、powdery→粉感……一次性敲定进仓库当常量。
2. **notes 气味词（MVP 范围内 100% 人工，含别名）**：Oud=乌木/沉香、Tonka Bean=零陵香豆、Ambroxan=降龙涎香醚。
3. **brand 品牌**：知名有公认中文名的用中文（Chanel→香奈儿、Dior→迪奥、Creed→信仰），小众沙龙保留原名更专业。
- 未命中一律兜底英文，不报错，可迭代补全。

## 7. 香水卡片设计

**字段总表（来源＋推导）**：

| 卡片字段 | 来源 | 推导规则 |
|---|---|---|
| 名/品牌/主图 | name/brand/picture | 品牌过映射表中文化 |
| 主香调（前 3–5） | accords 取 strength 最高前 N | 过香调表中文化，按强度降序（“柑橘·清新辛香·绿叶”） |
| 前/中/后调 | notes.tiered（或 flat） | 各过 notes 表；flat 则只显“主要气味”一行 |
| 适合季节 | seasons 四季计数 | 转占比 → 主季＋次季标签（见第8节平滑） |
| 时段倾向 | daypart 计数 | 占比 >60% 偏日/偏夜，否则“日夜皆宜” |
| 扩散力 | sillage_avg(1–4) | **映射 4 档枚举**（贴肤/近身/半臂/满室），不按小数切 |
| 留香 | longevity_avg(1–5) | 映射区间档（半天/一个白天/可能到夜晚），**不报小时数** |
| 浓度 EDP/EDT | **无此字段** | 从 name 正则弱推断并注“据名称推断”，否则标“浓度未知” |
| 风格标签 | accords＋notes＋daypart 规则 | 见下 |
| 性别倾向 | 官方 gender | 弱化展示，仅参考 |
| 评分/热度 | rating_avg/popularity | 评分需投票达阈值才显，热度仅用于排序 |

> **信息密度原则**：默认只露“主香调＋适合季节＋扩散＋留香＋风格标签”五项，前中后调/调香师/年份折叠进详情。避免工具感。

**风格标签规则（accords 强度＋daypart，可多标取前 2–3，阈值均为可调参数）**：清新通勤（citrus/aquatic/green ≥60 且 sweet<40 且 day占比≥55%）｜暖甜约会（sweet/vanilla/amber ≥55 且 night占比≥50%）｜正式商务（woody/aromatic ≥55 且 sweet<35 且 sillage 2–3）｜浓郁夜场（sillage_avg≥3.2 且 sweet/amber/oud/leather≥55）｜盛夏轻盈（summer占比≥40% 且 citrus/aquatic/green≥50）。规则化的好处：**可解释**（能告诉用户“因为它柑橘 100、白天票多，所以标清新通勤”）、零成本、可离线预算。

## 8. 推荐逻辑设计

设计哲学：**真实投票占比做基底，实时天气做乘性二次校准，所有对外表达都是区间/档位/人话，绝不输出可被证伪的精确数字。**

### 8.1 季节适配分（采纳红队④：必须平滑）

```
原始占比 season_fit[s] = seasons[s] / Σseasons
```
**问题**：低票香可能仅因 3 个夏天投票就 summer 占比 100%，会因“碰巧占比高”冒头。**修正——拉普拉斯平滑**：
```
season_smooth[s] = (seasons[s] + α) / (Σseasons + 4α)    // α=5
```
当前季节由日期＋实时气温共同确定（不只看月份）：temp≥28 偏 summer；temp≤8 偏 winter；10–22℃ 为 spring/autumn；湿度≥75% 强化 summer 的轻盈诉求。`S_season = season_smooth[current_season]`。

### 8.2 时间适配

`day_ratio = daypart.day / (day+night)`；白天用 day_ratio，夜晚用 1-day_ratio。

### 8.3 天气修正（采纳红队④：统一为乘性系数，删除 C 的独立加权项）

> **两套天气模型不能并存。最终统一采纳乘性 W**——天气是对“匹配度”的修正而非独立维度，符合物理直觉。

高温高湿放大甜/重/树脂/动物气息、易致闷腻；柑橘/水生/绿叶在热湿天更通透。冷干天清淡香“喷了像没喷”。体感档位（来自实时 API）：`hot_humid`(temp≥28 且 humid≥65) / `hot_dry` / `mild` / `cold`。

对每款香按其 top_accords 算乘性系数 W（封顶 0.7–1.3）：

| 体感 | 惩罚 accord（W↓） | 奖励 accord（W↑） |
|---|---|---|
| hot_humid | sweet/vanilla/amber/resinous/animalic/oud/heavy woody（命中且≥50，×0.85） | citrus/aquatic/green/aromatic/fresh（×1.15） |
| hot_dry | 重甜重树脂（×0.92） | citrus/dry-woody/aromatic（×1.1） |
| cold | 过分清淡/aquatic（×0.9，冷天易飘没） | amber/spicy/vanilla/resinous/leather（×1.15） |
| mild | ×1.0 | — |

### 8.4 综合排序

```
score = S_season^a × S_time^b × W × quality_prior
quality_prior = 贝叶斯收缩(rating_avg, vote_count)   // 低票降权
a=1, b=0.6（可调）
```
取 Top1 ＋ 2–3 个备选（优先来自 similar 或库内同风格）。一键换任意一瓶后，下面“怎么用”层照常重算。

> 维度子分 `breakdown` 摊给用户看（“夏季 +0.8，因夏季票占比 71%”），可信来自“经得起追问”。

### 8.5 喷量（sillage＋longevity＋环境 → 档位，绝不给 ml）

基础起点：sillage_avg≥3.2 → 1–2 下；2.4–3.2 → 2–3 下；<2.4 → 3–4 下。再按情境调：密闭/高密度（电梯/会议/医院/餐厅/地铁）**减一档**＋“宁少勿多”；开阔户外 +1；longevity≤2.5 提示“随身少量补喷”；longevity≥4 提示“很持久，少喷即可”。

### 8.6 喷洒位置

脉搏点（手腕/颈侧）扩散快留香强但“冲”；衣物/发梢柔和含蓄。想低调→衣物内侧（会议/医院）；想气场→颈侧/胸口（日常）；想留尾迹→手腕＋颈侧＋发梢少量（约会/晚宴）。

> **删除伪科学（采纳红队⑤）**：不写“揉搓破坏前调”（摩擦只是物理加速挥发），改无争议的“手腕高温会加速挥发，夏天可移到衣物”。保留“别直喷浅色丝绸/珍珠/皮草”这类真礼仪。

### 8.7 社交距离（采纳红队⑤：绑四个投票档位，不按小数切）

sillage 本就是四档投票，天然四分，直接绑档、不假装两位小数分辨率：

| sillage 档 | 人话 | 含义 |
|---|---|---|
| 档1（贴肤 intimate） | 贴近才闻得到 | 一臂内才察觉，适合密闭场合 |
| 档2 | 一臂之内 | 正常社交距离可感，安全日常 |
| 档3 | 一桌之间 | 同桌/小房间能闻到，注意场合 |
| 档4（外放 loud） | 整间屋都是它 | 电梯/会议慎用，户外/夜场加分 |

### 8.8 留香（longevity 1–5 → 区间，绝不给小时数）

1.0–2.0 较快淡去约半天（午后可补）｜2.0–3.0 大半个白天（傍晚有场合可补）｜3.0–4.0 基本撑一整个白天（无需补）｜4.0–5.0 可能延续到夜晚（少喷即可）。**戒律：绝不出现“6.2 小时”。**

### 8.9 风险提示（软提醒＋给替代，决策权在用户）

| 风险 | 触发 | 文案示例 |
|---|---|---|
| 扩散过强 | sillage档4 且 场景∈{电梯,会议,医院,餐厅,地铁} | “空间封闭、人多，这瓶气场偏大，建议只喷 1 下或换贴肤的。” |
| 闷甜不适 | sweet/amber≥55 且 hot_humid | “今天又热又潮，它偏甜重，戴久可能发腻，换清爽些的？” |
| 季节错配 | **主季占比 − 当前季占比 ≥ 阈值**（采纳红队⑤，改相对差，不用绝对15%） | “这瓶大家更多在秋冬用，今天用它会有点反季。” |
| 数据不足 | 在 MVP 精选集外/低票 | “这瓶社区数据较少，以下用法仅供参考。” |

## 9. 用户情境输入设计

反推原则：**只问“会改变用香建议”的维度，能自动获取的绝不让用户填，能一键选的绝不让用户打字。**

- **自动获取（零操作，高级感来源）**：天气/气温/湿度/风（QWeather＋定位，**绝不让用户拖滑块**）；时间/时段（本地时间）；季节（日期＋实时气温推断，不只看月份）。

- **一键 chip（真正必要的人为维度）**：① **场景**（唯一必须，决定喷量/位置/风险）：通勤/会议/约会/聚会/正式晚宴/居家/运动/就医；② 室内/户外（可选，多数从场景推断，影响喷量±1）；③ 想要的存在感（可选三档：低调/自然/想被注意到）。

- **自然语言入口（DeepSeek 解析，锦上添花非必填）**：“今晚和客户吃饭，餐厅不大”→ 正式晚宴＋室内密闭 → 减一档＋注意扩散。

> **MVP 取舍（采纳红队）**：自然语言输入框是额外故障点，**MVP 可砍，chip 选场景已够**，推到 v2。

- **用后反馈（必须有，否则只是机械推荐）**：每次给完用法后，下次打开或当晚轻问一句。三档情绪＋可选轻标签：`今天喷得` 太多/刚好/太少 →校准个人喷量基准；`留香` 短/差不多/久；`场合` 合适/偏冲/偏淡。**这些反馈只对该用户该香水做个人化偏移，不污染全局投票**——是“这瓶怎么用”长期壁垒的数据来源。

## 10. 前端体验设计

**核心词——分寸感**。设计目标：精致、简约、高级、可信、有美感，像“懂香水懂场景懂表达的智能体”而非查询工具。

### 10.1 信息架构（三 Tab，扁平）

底部导航（移动）/侧边（桌面）只放三项：**今日 · 香水库 · 我的**。复杂度收进每个面内部。

- **今日（默认首屏，传播入口）**：情境条（地点·天气·温湿度·时段，自动填，可点开编辑）＋今日推荐卡（主角）＋「换一瓶」＋反馈入口。
- **香水库（香柜）**：卡片网格＋筛选排序＋常驻「+添加」＋角标（今日是否被推荐/很久没用）。**添加＝搜名秒加抽屉**（中英混搜→点「+」即入库，无二级确认，搜不到则“手动记下名字”兜底）。
- **我的**：偏好画像（人话呈现系统学到的偏好）＋用香记录（反馈时间流）＋设置。

### 10.2 今日卡＋用香建议（采纳红队⑦：MVP 合并为卡片展开，砍一层跳转）

今日卡一屏给完结论：哪瓶＋一句话为什么＋三个关键结论（喷 2 下·贴肤型·半臂距离）。**点击展开即详情**（喷多少/喷在哪/社交距离/大约留香/适合此刻吗/风险提醒/为什么是这些建议——折叠默认收起）。换瓶后整页建议带 240ms 交叉淡出重排，让“换了→用法跟着变”可被看见。

### 10.3 可信化表达（三策略）

1. **证据可视化**：香调匹配条（accords 0–100 渲染为横向条）、季节匹配点（当前季高亮）、扩散档位条（sillage 四档）。所有条 0→实际值一次性增长 ≤600ms，不循环跳动。
2. **区间而非伪精确**：留香给“上午到午后/大约半天”不给“6.2 小时”；喷量给“1–2 下”附“先少后补”；距离给“贴肤到半臂”。
3. **“为什么”始终在场但不喧哗**：DeepSeek 解释引用真实情境＋真实数据＋分寸判断；数据不足诚实降级。

### 10.4 反馈交互（≤10 秒）

三档情绪 😕淡了点 / 🙂刚好 / 😣太冲了 ＋ 可选轻标签 [太快没味] [被同事注意到] [太甜] [太正式]。提交即时微反馈“记下了，下次帮你少喷半下”，累积后在偏好画像体现“你 3 次说太冲，我已默认调低扩散建议”。

### 10.5 设计语言（token 级，采纳红队⑧：向当代冷淡校正一档）

- **配色**：暖中性底（`--bg:#F7F5F1` 宣纸暖白 / `--surface:#FFFFFF` / `--ink:#1E1B17` 近黑暖棕）＋单一沉静主色（`--brand:#3E5C4B` 墨绿）＋低饱和状态色（`--warn:#B8895A` 暖琥珀作分寸提醒，不用刺眼红；`--accent:#C8A96A` 香槟金仅极少强调）。每瓶香取一个主香调色点作卡片点缀。暗色模式 v2。
- **字体**：中文标题思源宋体（差异化、手账感），正文思源黑体（可读）；英文衬线（Spectral）配香名，数字 tabular-figures 对齐。
- **间距/圆角/阴影**：8pt 间距阶梯，卡片圆角 16px、标签胶囊 999px；阴影极淡 `0 8px 24px rgba(30,27,23,.06)`（“几乎看不见但有浮起感”）。
- **动效**：只在三处用——卡片入场（淡入+8px 上移 200–280ms）、证据条增长、换瓶重排。禁循环动画/视差/弹跳。
- **视觉校验**：宋体标题保留作差异化，但整体向“当代冷淡/高级灰/欧美杂志感”校正一档，避免全压水墨东方的自我感动。

### 10.6 留存（不靠推送轰炸）

每日变化即理由（天气/时段/场景驱动）；轻记录累积感（连续天数弱标记、季节切换“入秋了，你库里这 3 瓶进入主场”、吃灰温柔提醒）；推送默认至多每日一条早间软提醒（可关）。

### 10.7 命名体系（品牌化术语，需与代码字段口径对齐一次）

香柜（库）｜今日一喷（今日推荐）｜换一瓶｜分寸建议（用香建议）｜「今天，刚好吗？」（反馈）｜此刻（情境）｜气味档案（详情）。

## 13. 数据结构（关键 TypeScript 类型）

```ts

interface Perfume {
  id: number;
  name: string; brand: string;
  brandZh: string;                 // 中文品牌（映射，兜底=brand）
  gender: 'male' | 'female' | 'unisex';
  year: number | null;
  rating: number | null;           // 1..5
  longevity: number | null;        // 1..5  1弱5持久
  sillage: number | null;          // 1..4  1贴肤 4张扬
  sillageTier: 1 | 2 | 3 | 4;      // 预计算的四档枚举（社交距离/喷量用，不按小数切）
  priceValue: number | null;       // 1..5
  seasonPct: { winter: number; spring: number; summer: number; autumn: number }; // 拉普拉斯平滑后占比
  daypartPct: { day: number; night: number };
  accords: { name: string; strength: number }[];  // name=中文, 0..100, 已按强度降序
  notes: string[];                 // 中文化香调词（top+middle+base 扁平）
  styleTags: string[];             // 预计算风格标签（清新通勤/暖甜约会…）
  popularity: number;              // 同分次级排序
}

// ── 13.2 用户香水库条目（localStorage / v2 Supabase）──
interface UserPerfume {
  perfumeId: number;
  addedAt: number;
  nickname?: string;
  bias?: {                         // 个人偏置，由 Feedback 聚合（早期全 0）
    likeScore: number;             // -1..+1 整体偏好微调
    perceivedStrength: number;     // -1..+1 觉得它比社区更冲(+)/更弱(-)
  };
}

// ── 13.3 情境（传感 + 解析合并后）──
interface Context {
  tempC: number; humidity: number; windSpeed: number; weatherText: string;
  feel: 'hot_humid' | 'hot_dry' | 'mild' | 'cold';   // 预计算体感档
  daypart: 'day' | 'night';
  season: 'winter' | 'spring' | 'summer' | 'autumn';
  occasion?: 'work'|'date'|'social'|'formal'|'casual'|'sport'|'commute' | null;
  formality?: number | null;       // 0..1
  intimacy?: 'close'|'neutral'|'broadcast' | null;
  avoid?: ('too_sweet'|'too_strong'|'too_formal'|'cloying')[];
  notePreference?: string[];
  rawText?: string;
}

// ── 13.4 推荐结果 ──
interface Recommendation {
  primary: ScoredPick;
  alternatives: ScoredPick[];      // 2~3 瓶
  context: Context;
  explanation: string;             // DeepSeek 生成（降级时为模板）
  generatedAt: number;
}
interface ScoredPick {
  perfumeId: number;
  score: number;                   // 0..1
  breakdown: Record<string, number>; // 各维度得分，摊给用户看（透明可信）
  usage: Usage;
  risks: string[];
}

// ── 13.5 用法（规则计算，产品壁垒层）──
interface Usage {
  spraysLabel: string;             // "2~3 下"，区间表达不伪精确
  placement: string[];             // ["手腕","颈侧"]
  socialDistance: 1 | 2 | 3 | 4;   // 绑 sillageTier，人话档位
  durationHint: string;            // "半天左右"，绝不给小时数
  suitable: boolean;
}

// ── 13.6 用后反馈（驱动 bias 个人偏移）──
interface Feedback {
  perfumeId: number;
  contextSnapshot: Pick<Context,'season'|'daypart'|'tempC'|'humidity'|'occasion'>;
  rating: 'too_weak' | 'perfect' | 'too_strong';   // 三档情绪
  tags?: ('faded_fast'|'noticed'|'too_sweet'|'too_formal'|'wrong_vibe')[]; // 可选轻标签
  createdAt: number;
}
```

打分核心（季节已平滑、天气乘性、权重显式可单测。**以下为实现现状**）：
```ts
// 线性项权重归一到 1（无 weather、无独立 preference 加性项）
const WEIGHTS = { season:0.41, daypart:0.19, occasion:0.40 };
const linear = WEIGHTS.season*b.season + WEIGHTS.daypart*b.daypart + WEIGHTS.occasion*b.occasion;
// 个人偏好走乘性通道（不占加性权重，避免与线性项口径混淆）：
const biasMul = 1 + likeScore * 0.25;              // likeScore∈[-1,1]
score = linear * W * qualityPrior * biasMul * avoidPenalty;
// weather 以乘性 W∈[0.7,1.3] 作用；avoidPenalty 命中场景规避则 ×0.6~0.85，不归零（保留可解释）
```
> 早期设计曾拟 `preference:0.12 / bias:0.08` 两条加性项；实现改为「preference 并入乘性 biasMul、无独立加性权重」，故此处权重与旧稿不同。

**判据**：一个全栈个人几周内交付，完整验证“先推荐＋可更换＋教你怎么用＋反馈＋发现型钩子”。

### 14.1 做什么（MVP 必须有）

| 模块 | 范围 |

## 15. 后续迭代方向

- **阶段一 · MVP——验证需求真不真**：跑通闭环，看有没有人每天回来。衡量：次日/7 日打开率、换香率、吃灰采纳率、反馈提交率。
- **阶段二 · 深化“怎么用”——构筑壁垒**：反馈飞轮从“调 sillage 偏移”扩到多维个人画像（怕甜/怕冲/某调性敏感）；穿搭/场景增强（图像识别）；香水库资产化（使用日历、季节轮换、扩到全量并补全中文化）。**叙事诚实降级**：早期不承诺“逐用户逻辑回归”，主线是“全局规则＋个人硬偏移”（用户说 3 次太冲就全局降一档扩散建议——简单、可感、不需大数据）；逻辑回归是数据攒够后的远期选项，不是 MVP 依赖项。
- **阶段三 · 扩展与商业化（守住非导购）**：“今天喷哪瓶”做成可分享卡片低成本拉新；**变现走订阅/报告**（个人香水画像报告、年度用香回顾、季节轮换指南），不碰交易、不种草。

## 16. 潜在风险与反驳

### 16.1 三个致命问题与对策（采纳红队“二、三大致命问题”）

- **致命1 · 核心痛点强度不足**：“今天喷哪瓶”是产品经理的痛、不是用户的痛（用户已有“喷固定瓶”零成本解法且不痛苦）。**对策**：主打改为发现型价值（吃灰提醒 S5＋天气突变预警 S4）；MVP 上线盯换香率/吃灰采纳率验真；面试讲“我在验证一个海外已初步验证、国内未满足的中频需求，并设计了证伪指标”——展示判断力比假装高频刚需得分。
- **致命2 · 壁垒对香水时代不成立 ＋ 反馈飞轮冷启动期转不动**：单用户一月几十条反馈分散到 8 瓶×4 天气×5 场景＝每组合 0.2 条，喂不饱模型；香水时代有用户有数据，2 周能抄“今天喷哪瓶”。**对策**：壁垒重新定义为**“商业模型冲突”**——香水时代做“帮你用好已有香水”＝抑制复购＝自伤 GMV，结构上不会做；冷启动壁垒换成“即时可感”（吃灰/天气预警/换香即重算）；反馈飞轮降维成“全局规则＋个人硬偏移”。
- **致命3 · 单人工作量 × 中文映射深坑**：“Top-N＋英文兜底”意味着大量卡片露英文，恰好踩中自己定义的生死线。**对策**：MVP 只精选 Top 1–2 千 **100% 中文化**，搜不到的让用户手动记名字（用“小而全中文”换“大而露英文”）；工作量诚实排序（先做最短可 demo 闭环，反馈/画像/暗色/相似香全推 v2）；作品集要“完整闭环＋清晰判断”不是“功能多”。

### 16.2 面试官 5 个尖锐质疑 + 反驳话术

- **Q：“用户随手喷一瓶根本不痛苦，伪需求吧？”** → “对，‘纠结喷哪瓶’是弱痛点，所以我没把它当核心卖点。真正价值是两个用户会主动认同的点：重要场合不翻车（有社交代价的真焦虑）、把吃灰好香用起来（真实的钱包痛）。而且我设计了证伪指标——换香率和吃灰采纳率，上线就能验证。”
- **Q：“香水时代两周能抄你，壁垒在哪？”** → “技术上它能抄，但它结构上不会做。它的模型是导购 GMV，‘帮你用好已买的’本质抑制复购、自伤收入。这不是能力问题是商业模型冲突。我占的恰是它因赚钱方式而主动放弃的位置。”
- **Q：“一个用户一月几十条反馈，喂得动模型吗？”** → “喂不动复杂模型，所以我不用。早期是‘全局规则＋个人硬偏移’——你说 3 次某类香太冲，我就对你直接降一档扩散，简单当场可感不需大数据。逻辑回归是远期选项，不是 MVP 依赖。”
- **Q：“数据全英文 Fragrantica，做中文不出戏？36,705 全翻了？”** → “没全翻，全翻是陷阱。MVP 只精选中国用户真会拥有的 Top 1–2 千，这部分香调品牌 100% 人工中文化，主流体验零英文；长尾搜不到让用户手动记名字。另外坦白：原始数据 vote_count 和 people 是两列、1.1 万行不等，所以精选基数按哪列筛略有差异（36,705/36,648），这种脏活我逐列核对过。”（主动暴露不一致反证严谨）
- **Q：“留香、社交距离凭什么说得准？拍脑袋编的？”** → “我刻意不说准。留香只给‘半天/一整天’区间不给‘6.2 小时’；社交距离绑 sillage 四个投票档位给‘一臂之内/整间屋’人话不给小数。所有量来自 Fragrantica 几十上百人真实投票均值，不是我编的字段。敢于不精确本身就是产品判断——无法验证的精确会摧毁信任。”

## 17. 最终推荐的优化版本（一页纸定稿）

**一句话定位**：氛寸——你已经有这些香水，它帮你决定此刻该喷哪一瓶、怎么喷得恰到好处。（传播版：别人帮你挑香水，氛寸帮你用好香水。）

**核心闭环**：
```
搜名秒加建香柜 → QWeather 自动感知此刻情境 → 规则引擎从你库里推一瓶
  → 不认同?一键换任意一瓶（用法即时重算）→ 拿到「分寸建议」(喷几下/喷哪/
  社交距离/留香区间/风险) → 出门 → 回来答「今天，刚好吗?」→ 系统个人偏移收敛
                ↑ 发现型钩子：天气突变预警 + 吃灰提醒（传播主线）
```

**技术栈定稿**：Next.js 15(App Router)+TS ｜ Tailwind+shadcn/ui ｜ Zustand ｜ **规则引擎前端本地打分（决策）+ DeepSeek 仅做解析与解释（表达，可降级）** ｜ MiniSearch 预建索引 ｜ localStorage(v2→Supabase，Storage 接口抽象) ｜ QWeather 服务端+30 分钟缓存 ｜ 构建期裁剪静态 JSON 上 Vercel CDN ｜ **不上向量库**。

**MVP 清单（几周可交付）**：① 搜名秒加（Top 1–2 千 **100% 中文化**）② QWeather 自动情境（城市兜底）③ 规则推荐（季节拉普拉斯平滑×时段×天气乘性 W×质量先验）④ 一键换香＋用法重算 ⑤ 分寸建议（喷量档/位置/社交距离四档/留香区间/风险，全程不伪精确）⑥ 场景 chip ⑦ **天气突变预警＋吃灰提醒** ⑧ 三档反馈→个人 bias 偏移 ⑨ 三 Tab 响应式（GitHub+Vercel）。**砍**：全量数据、ML 逻辑回归、自然语言框、偏好画像、用香日历、相似香、暗色模式、独立详情页、种草变现——全推 v2。

**3 个面试记忆点**：
1. **“我做的不是香水导购，是香水的‘天气预报’。”**——每天早晨基于真实天气体感，从你已有香水里告诉你今天喷哪瓶、喷多少。
2. **“国内所有香水产品都停在‘买’，我做了‘买之后’。”**——香水时代做导购、嗅氪做定制，没人管“你已有的香水今天怎么用”。**真壁垒是商业模型冲突：它做这个＝抑制复购＝自伤 GMV，结构上不会做。**
3. **“我把约 3.6 万款真实社区投票香水接上实时天气和 DeepSeek，做成一个有分寸感的智能体——且我刻意不伪精确（不给‘留香 6.2 小时’），因为无法验证的精确会摧毁信任。”**——数据规模、技术栈、产品克制一句秀全。

> 一句收尾把壁垒钉死：**“形态可以被抄，但‘每天清晨那一句刚好的提醒’背后、香水时代因商业模型冲突而永远不会做的那个位置，抄不走。”**
